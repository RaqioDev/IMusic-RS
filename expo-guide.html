Idea central

En una frase

Los sistemas de recomendación actuales nos muestran siempre lo mismo porque solo miran el historial fijo y no escuchan cómo cambia nuestro gusto según el momento. Este proyecto diseña uno que sí lo hace.

Problema 01

Perfil estático vs. gusto dinámico

El sistema te conoce de hace 6 meses. Pero hoy es lunes a las 7 AM y querés algo enérgico para ir al gimnasio. La plataforma igual te pone boleros.

El problema: el modelo no sabe que tu gusto cambia con el contexto.

Problema 02

Señales implícitas sin explotar

Cada vez que saltás una canción a los 10 segundos estás diciendo "no me gusta". Cada vez que la escuchás tres veces seguidas estás diciendo "me encanta".

Los sistemas actuales tiran esa información por el volumen que genera.

Problema 03

Cold start — el usuario nuevo

Un usuario que se registra hoy no tiene historial. El sistema no sabe nada de él y no puede recomendar nada útil.

Solución propuesta: arrancar desde las características de la canción en lugar del historial.

¿Por qué cuatro papers respaldan esto?

Cuando presentás la sección de referencias, explicá para qué sirve cada una. No hace falta memorizarlas: simplemente decís qué problema resuelve cada autor.

Guión sugerido para arrancar

"Todos usamos Spotify, YouTube Music o alguna plataforma musical. ¿Les ha pasado que el sistema les recomienda lo mismo siempre, aunque a veces quieren algo completamente diferente? Eso pasa porque estos sistemas no saben nada del contexto: qué hora es, qué están haciendo, cómo se sienten. Nuestro proyecto diseña una arquitectura que sí tiene en cuenta eso — usando datos de comportamiento en tiempo real y características de las canciones mismas..."

Glosario — términos que van a aparecer y cómo explicarlos
Sistema de recomendación
Algoritmo que predice qué le va a gustar a un usuario basándose en datos previos. Netflix, Spotify y Amazon lo usan.
Cold start
El "problema del usuario nuevo": si no hay historial, el sistema no sabe qué recomendar. Es uno de los retos más conocidos del área.
Señales implícitas
Acciones que el usuario no hace conscientemente para opinar, pero que revelan preferencias: saltarse una canción, escucharla entera, pausarla.
Features acústicas
Características medibles de una canción: qué tan rápida es (BPM), qué tan energética, qué tan "alegre" suena, si tiene instrumentos acústicos, etc.
Modelo híbrido
Sistema que combina dos métodos de recomendación: uno basado en usuarios similares y otro basado en las características de las canciones.
Pipeline
Secuencia de pasos encadenados donde la salida de uno es la entrada del siguiente. Como una línea de producción en una fábrica.

Tu sección en la presentación

¿Qué tocás vos?

  • La tarjeta de "Arquitectura técnica" (stack tecnológico)
  • El pipeline de 5 fases — presentás las primeras tres (I, II, III)

Idea central

En una frase

El sistema captura cada interacción del usuario, la enriquece con los datos de la canción y construye un vector matemático que representa el "sabor musical" de ese oyente. Luego agrupa oyentes similares.

Stack tecnológico — por qué estas herramientas

Tres componentes clave

PySpark + MLlib Motor de procesamiento distribuido. Permite procesar millones de canciones y usuarios al mismo tiempo, como si tuvieras cientos de computadoras trabajando juntas.
HDFS / Parquet Sistema de almacenamiento distribuido y formato de archivo comprimido. Guarda los datos en muchos servidores a la vez y los comprime hasta 8 veces para que ocupen menos.
Spotify Dataset (Kaggle) Dataset público con 114,000 canciones y sus características acústicas ya calculadas. Es el insumo de datos para todo el sistema.
Fase I — Modelado de señales de entrada

¿Qué datos captura el sistema?

Antes de procesar nada, el sistema necesita saber qué le está diciendo el usuario con cada acción. No todas las acciones tienen el mismo peso.

Señales explícitas (el usuario sabe que opina) Guardar una canción, agregarla a una playlist, seguir un artista. Son pocas pero de alta confianza.
Señales implícitas (el usuario no sabe que opina) Saltar una canción a los 10 segundos = "no me gusta". Escucharla completa tres veces = "me encanta". Son muchas y muy valiosas.
Señales de contexto (el cuándo y el dónde) Hora del día, día de la semana, tipo de dispositivo, geolocalización aproximada. Permiten saber si el usuario está en el gym, trabajando o relajándose.
Formato de registro Cada interacción se guarda como: usuario, canción, tipo de evento, cuándo ocurrió, desde qué dispositivo, cuánto duró.
Fase II — Ingesta y enriquecimiento acústico

¿Cómo se procesan y enriquecen los datos?

Logs de usuario (JSON) → HDFS / Parquet → PySpark JOIN → Dataset enriquecido

El "enriquecimiento" consiste en cruzar los eventos del usuario con los datos de cada canción. El resultado es una fila que combina: qué hizo el usuario + cómo suena esa canción.

Fase III — Perfilado de usuario con K-Means

¿Cómo agrupa el sistema a los oyentes?

Con los datos enriquecidos, el sistema calcula un "vector de perfil" para cada usuario: el promedio de las características de las canciones que escuchó completas, con penalización por las que saltó.

¿Qué es K-Means? Un algoritmo que agrupa puntos parecidos en K grupos. Aquí: agrupa usuarios con gustos similares. Si dos usuarios tienen vectores de perfil parecidos, van al mismo cluster.
⚡ Cluster 1 — Oyente energético Alta energía, alto BPM, alta bailabilidad. Probablemente escucha en momentos de ejercicio o desplazamiento.
🌙 Cluster 2 — Oyente contemplativo Baja energía, alta acusticidad, alta instrumentalidad. Noche, estudio o relajación.
🎷 Cluster 3 — Oyente de nicho Mezcla de features, escucha concentrada en fines de semana. Alta instrumentalidad.
Guión sugerido para tu parte

"Una vez que entendemos por qué el problema existe, vamos a ver cómo el sistema lo ataca. Todo empieza por los datos. Cada vez que escuchamos música estamos generando información: cuándo pusimos play, cuándo saltamos la canción, desde qué dispositivo. El sistema captura todo eso, lo combina con las características musicales de cada canción — qué tan energética es, qué tan alegre suena — y construye un 'retrato musical' de cada oyente. Después agrupa oyentes parecidos usando un algoritmo llamado K-Means, que es básicamente como organizar personas en grupos según sus preferencias..."

Glosario — términos técnicos para tu sección
K-Means Clustering
Algoritmo que divide un conjunto de datos en K grupos (clusters). Pone en el mismo grupo los puntos que son más parecidos entre sí. En este caso, agrupa oyentes con gustos similares.
Vector de perfil
Representación matemática del gusto musical de una persona como una lista de números. Ejemplo: [BPM:0.8, Energy:0.9, Valence:0.7...]. Permite compararlos matemáticamente.
PySpark
Herramienta de programación que permite procesar enormes volúmenes de datos distribuyendo el trabajo entre muchos procesadores a la vez. Como dividir una tarea entre muchas personas.
HDFS / Parquet
HDFS es un sistema de archivos distribuido (como un disco duro compartido en la nube). Parquet es un formato de archivo muy comprimido y eficiente para análisis de datos masivos.
Método del codo
Técnica gráfica para elegir cuántos clusters (K) usar en K-Means. Se busca el punto donde agregar más clusters deja de mejorar significativamente los resultados: el "codo" de la curva.
Enriquecimiento acústico
Proceso de unir los datos de comportamiento del usuario (qué hizo) con las características musicales de cada canción (cómo suena). El resultado es mucho más informativo que cada fuente por separado.

Idea central

En una frase

El motor combina dos estrategias según cuánto sabe de vos: si tenés historial, te compara con usuarios similares; si sos nuevo, analiza directamente las canciones que conocés. Y lo medimos viendo si saltás menos canciones.

Fase IV — El motor híbrido

¿Cómo decide el sistema qué recomendarte?

El motor usa dos enfoques al mismo tiempo y los combina según cuántos datos tiene del usuario. Esto resuelve el problema del cold start (usuarios nuevos) sin sacrificar la calidad para usuarios con historial.

Filtrado colaborativo — ALS "Usuarios que escucharon lo mismo que vos también les gustó X".

Usa un algoritmo llamado ALS (Alternating Least Squares) que descompone una matriz enorme (usuarios × canciones) en patrones latentes. No mira el contenido de la canción: solo el comportamiento de oyentes parecidos.

Requiere historial suficiente para funcionar bien.
Filtrado por contenido "Esta canción suena parecida a las que te gustaron".

Compara el vector acústico de canciones que le gustaron al usuario con todo el catálogo usando similitud coseno (qué tan parecidos son dos vectores matemáticamente).

Funciona incluso sin historial: solo necesita una canción de referencia.

Historial del usuarioEstrategiaPeso colaborativo / contenido

Fase V — Evaluación: ¿cómo sabemos si funciona?

Las métricas que mediríamos

Como es una propuesta teórica, no ejecutamos la evaluación, pero la diseñamos rigurosamente. Eso es lo que diferencia un proyecto serio de una especulación. Estas son las tres métricas clave:

↓ Skip
Porcentaje de canciones saltadas antes de los 30 segundos. Un buen sistema produce menos skips. KPI primario.
SKIP RATE — KPI PRIMARIO
↑ Save
Canciones recomendadas que el usuario guarda en su biblioteca. Indica que la recomendación tiene valor a largo plazo. KPI de engagement.
TASA DE GUARDADO — KPI ENGAGEMENT
P@K
De las primeras K recomendaciones, ¿cuántas son realmente relevantes para el usuario? Métrica estándar en recuperación de información. KPI de calidad.
PRECISION@K — KPI CALIDAD
Datasets para validar

¿Con qué datos se evaluaría?

  • Spotify Million Playlist Dataset — 1 millón de playlists públicas. Permite comparar las recomendaciones contra playlists reales como "respuesta correcta".
  • Free Music Archive (FMA) — dataset abierto con features acústicas pre-calculadas. Ideal para la rama de filtrado por contenido.
Protocolo de evaluación A/B

¿Cómo se compararía?

Se divide el dataset en dos grupos:

  • Control: recomendación por popularidad global (el enfoque más simple)
  • Experimental: motor híbrido diseñado en este proyecto

El éxito se define como: menos skips y más Precision@K en el grupo experimental.

Guión sugerido para cerrar

"Hasta acá entendimos el problema y cómo se procesan los datos. Ahora viene la parte donde el sistema realmente hace las recomendaciones. La clave es que no usa un solo método, sino dos: uno que dice 'a usuarios parecidos a vos les gustó esto' y otro que dice 'esta canción suena parecida a las que te gustaron'. Cuánto peso le da a cada uno depende de cuánto sabe de vos. Si sos usuario nuevo, confía más en la música. Si tenés historial largo, confía en la comparación con otras personas. Y para saber si funciona, medimos cuántas canciones saltás: si saltás menos, el sistema está recomendando mejor..."

Glosario — términos técnicos para tu sección
ALS — Alternating Least Squares
Algoritmo que descompone una tabla enorme (usuarios × canciones) en dos tablas más pequeñas de "factores latentes". Permite encontrar patrones ocultos de gusto sin leer las canciones.
Filtrado colaborativo
Recomienda en base a lo que hicieron usuarios similares. "A tus vecinos musicales les gustó esto, entonces probablemente a vos también." No mira el contenido de la canción.
Filtrado por contenido
Recomienda canciones con características similares a las que ya te gustaron. "Esta canción tiene el mismo ritmo y energía que tus favoritas." No necesita historial.
Similitud coseno
Medida matemática que calcula qué tan parecidos son dos vectores (listas de números). Si dos canciones tienen vectores similares, suenan parecido. Va de 0 (opuestos) a 1 (idénticos).
Precision@K
De las primeras K recomendaciones que el sistema muestra, ¿cuántas son relevantes? Si K=10 y 7 son buenas, Precision@10 = 0.7. Métrica estándar en sistemas de búsqueda y recomendación.
Skip Rate
Porcentaje de canciones que el usuario salta antes de los 30 segundos. Un skip temprano indica que la recomendación fue mala. Reducirlo es el objetivo principal del sistema.
